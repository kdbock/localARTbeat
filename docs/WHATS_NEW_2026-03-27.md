# What's New

Date: March 27, 2026

This note summarizes the concrete follow-through completed while working
through the investor audits, and separates that from the material that still
remains open.

## Major Progress Since The Audit Pass

### Foundation And Architecture

- Root `lib/` was formalized as the application shell and composition layer.
- `artbeat_core` was reduced from a cross-feature host into a cleaner base
  package with no sibling feature-package dependencies.
- Major cross-package dependency cleanup landed across profile, admin, auth,
  messaging, settings, ads, capture, community, art-walk, artist, and artwork.
- Routing ownership was pushed upward into the app shell instead of remaining
  spread sideways across feature packages.

### Startup And Runtime Integrity

- Global error handling was centralized into one bootstrap path.
- Provider-owned services now rely more on explicit `initialize()` ownership
  instead of hidden constructor side effects.
- Startup integrity rules were documented in
  `docs/STARTUP_INTEGRITY_STANDARD.md`.
- Constructor-safety regression tests were added for startup-critical services
  in core, messaging, capture, art-walk, and sponsorships.

### Release Confidence

- The targeted release-confidence manual QA pass completed successfully on
  March 26, 2026.
- Account/profile persistence passed.
- Paid local ad and sponsorship review flows passed the targeted release
  checks.
- Capture upload reliability passed.
- Chat media upload reliability passed after shared upload hardening and
  Storage rule alignment.
- Admin data-rights deletion handling passed.

### Security And Configuration Hygiene

- Stripe configuration handling was tightened for release builds.
- Release builds now require explicit release Stripe key injection instead of
  relying on runtime `.env` defaults.
- Cloud Functions base URL resolution now supports environment-driven service
  discovery.
- CI now includes high-risk secret scanning and fails when tracked sensitive
  files such as `.env.production` or `debug.keystore` are present.
- Firebase Functions checks were added to CI.

### Documentation And Operating Discipline

- Canonical operating docs were consolidated under `docs/`.
- Active work now has one primary queue in `docs/WORK_QUEUE.md`.
- Key decisions are tracked in `docs/DECISIONS.md`.
- The current dependency graph is tracked in
  `docs/PACKAGE_DEPENDENCY_INVENTORY.md`.

### Localization

- English is now treated as the canonical locale source.
- Translation key parity reporting is automated.
- Current locale files now report zero missing keys against `en.json`.
- Translation cleanup has moved from missing-key repair to untranslated text
  and hardcoded-string cleanup.

## Audit Areas With Meaningful Updates

### Execution Confidence

Most improved area.

- Startup ownership is clearer.
- Route handling is more decomposed.
- Service initialization contracts are documented and tested.
- Release-confidence QA evidence is current and passing.

### Risk Profile

Meaningfully improved, but not fully closed.

- Client payment/config handling is safer than it was during the audit.
- CI now enforces more hygiene around tracked sensitive files and secrets.
- Production-like legal/data-rights work has current staging and QA evidence.

### Scalability Of Team And Process

Improved, especially in architecture and CI.

- Package boundaries are materially cleaner.
- CI coverage now includes ads, admin, sponsorships, and functions.
- Project-control docs are more systematic than during the audit snapshot.

## Areas Still Open

### Defensibility

- Route analytics still look like route/popularity telemetry, not a moat.
- No clear evidence yet of recommendation loops, creator quality scoring,
  sponsor ROI intelligence, or ad yield optimization.
- The moat hypothesis still needs to be written down explicitly.

### Marketability

- Funnel instrumentation is still weaker than investors will want.
- No clear single monetization catalog or pricing experiment layer is visible.
- No investor-facing KPI reporting layer is visible yet for activation,
  conversion, retention, and revenue by stream.

### Process And Ownership

- No visible `CODEOWNERS` file yet.
- Package ownership rules are not yet formalized enough for a larger team
  model.
- Some orchestration files remain larger than they should be.

### Release Completion

- The main remaining release gate is the legal/data-rights production canary.

## Practical Next Implementation Order

1. Finish the remaining execution-confidence cleanup in route/provider-owned
   services and large orchestration files.
2. Continue translation cleanup using the generated untranslated-text report.
3. Add funnel-level monetization and activation telemetry.
4. Formalize ownership/process guardrails such as `CODEOWNERS` and package
   ownership rules.
5. Execute the production legal/data-rights canary before final release work.
